modalities
Throughout this talk, I'll refer to each of these as - "modalities"; with a few exceptions, most software makes you choose one or the other in order to create a visual representation of data
Why do we care? - Consider this infographic that we found in the wild (some of you have probably seen me use this example before); the overall point of the visualization is to show that HBO reuses many actors across many of their shows; actors are listed on the left alphabetically, and shows are listed on the right alphabetically, and a line indicates that an actor acted in that show. - There's an interesting artifact at the bottom, where one actor, J.D. Williams is out of alphabetical order - and the designer knows about it, because there's a written apology at the bottom. Think about which modality the designer probably used to create this chart. He probably drew the thing in something like Illustrator, and then someone gave him a new data point. We can see a tension here between automation and drawing: our designer friend couldn't have used something like Excel or Tableau to create this... not just because one of the actors is out of order, but because you can't create this kind of layout in an automated way without writing code. By using a drawing tool to create it, he's committed himself to the layout, and has to deal with drawing each point one at a time by hand. If he were to have written code to create this chart, it would be easy to add new data, or even do things like change the sort order or try different layouts. But using automation would mean that some things—like placing the exact x,y coordinates of the title at the top, experimenting with typography or the color scheme, or even getting the margins right—would be tedious. And all this assumes that our designer has the time to learn how to code. - Here's another example. Here, we only see five data points, so scalability isn't as much of a concern—most of the work that would go into something like this is placing the text and getting the colors right... so a drawing program would probably be a better choice, right? Of course, if you look closely, you'll notice that the data shown in the pies doesn't match the data shown in the text. There are two 85% slices, and they're different sizes... and the 81 and 52 slices are definitely wrong. This is another example of the tension between automation and drawing; most of the work in creating this chart is best done in a drawing program, except that getting thigns precise can be a little tricky... an automated approach kind of gives you precision for free.

iteration
We're focused on a visualization designer's workflow - which is usually guided by a process model like the d.school's "emphathize, define, ideate, prototype, test" - or the vis community's 9-stage design study methodology - These process models encourage iteration as a critical component; it's important to go back to earlier stages in your workflow, especially when you discover something new This means that a designer's exact workflow—or someone's precise path through this space—is impossible to predict. It would be really inappropriate to try to anticipate or prescribe a precise linear path through these phases.

Related Work

Automated Visualization Toolkits

  • Formal specification of rules, computer does the drawing
  • Examples: D3, Processing, Excel, Tableau, Gephi
  • Pros:
    • Create precise visual representations of data efficiently
    • Scalable; time spent does not depend on data size
  • Cons:
    • Every graphics element must be formally specified,
      either by the user... or the toolkit developer
    • Designers don't use them in isolation

Drawing Software

  • (w.r.t. data:) informal rules, human does the drawing
  • Examples: Illustrator, Photoshop, Inkscape
  • Pros:
    • User is free to draw what they want
    • Direct manipulation faster, easier than formal specification
  • Cons:
    • Precision
    • Scalability

How do we know?

What we saw

  • Designers did not necessarily dislike manual encoding
  • The cost of manual encoding prevented variation
  • Manual encoding was a form of data exploration
  • Data exploration & manipulation related to experience
  • High-level elements, followed by sketching in data points
  • Encoding was a later consideration
Notes

How do we know?

What we saw

  • Inferences about behavior were often inaccurate
  • Encodings remained unchanged until assumptions were shown to be incorrect
  • Inferences about structure were often separated from actual data
  • Designers did not vary from given data structure
  • Designers disliked tools that enforced a workflow
  • Automation still resulted in manual labor

How do we know?

Opportunities

  • Manual encoding has its benefits
  • Placing data on existing graphics
  • Relaxing the sequence of operations
  • Creating an effective data abstraction

Bridge Model

Iteration with automated visualizations

Notes
Ideally, let's pretend we have a visualization in an automated tool, that I'm just going to represent with this abstract circle - We'd want to be able to open it in a drawing program - make some changes in the drawing program - and move it back, with the changes, into the automated program - where we could make more changes - and so on and so forth. Unfortunately, we can't do this right now; current automated visualization tools don't let you edit existing graphics.
Part of the reason is that our picture is a little simplistic; from an authoring perspective, a visualization is more than just graphics. - Each modality comes with extra, non-transferable elements that only exist in one tool. - For example, on the drawing side, a tool might provide color swatches, layers, or typography support that you can't get anywhere else - An automated tool has formalized rules that make it possible to manipulate graphics in a scalable way, as well as unique features of their own, such as force-directed layouts
So how do we make this work when the features that we need are spread across multiple programs? The core idea of the bridge model is to make changes in parallel, and merge their effects on the graphics. So to repeat the earlier workflow, we start with an automated visualization, - copy it into the drawing program - make drawing changes - make automated changes - merge the results - and bring the merged result back into each tool
In practice, it might look something like this. Let's pretend that I'm creating that HBO infographic from earlier. - I might start by generating the layout with D3 - and then exporting that to Illustrator, - where I change the colors and typography - (zoom in) Now if we zoom in, we can see that the actors are sorted first name... what if I wanted to sort by last name? - (zoom out) - (merge) I would merge my color changes back over to D3 - (reintegrate) (zoom in) - where I can change how they're sorted - (zoom out) Now let's imagine that our client comes to us, and they want the graphic to fit in a square instead of this letter-sized page. - (merge) If we merge back to the Illustrator, - (reintegrate) it's easy to crop off the bottom of the page... but we're also going to need a new layout because the bottom part of our graphic is going to get cut off - (clean) This time, let's make the change on both sides in parallel. - So we can crop the page on the right using Illustrator - We can implement a new layout on the d3 side, - and merge the changes! - Let's reintegrate the results... - (clean) And now that title is kind of in the wrong spot... - So let's move it with Illustrator - (merge) And merge that back over to d3 - (reintegrate)
